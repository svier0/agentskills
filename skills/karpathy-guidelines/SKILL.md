# 卡帕西准则

减少大语言模型常见代码编写错误的行为准则, 源自[安德烈·卡帕西的相关观点](https://x.com/karpathy/status/2015883857489522876)，内容围绕大语言模型代码编写相关隐患展开.

**Tradeoff:** 这些准则优先侧重稳妥审慎，而非追求效率。处理简易任务时需酌情灵活把控。

## 1. 编码前先思考

**切勿主观臆断，不要掩饰疑问，直面利弊取舍。**

实施前:
- 明确列出你的各项假设。存有疑问时及时提问。
- 若内容存在多种解读方式，全部列明，不得擅自选定其中一种。
- 若有更简便的解决思路，如实说明；必要时提出异议。
- 遇到表述模糊的内容即刻暂停，指明疑点所在并发起询问。

## 2. 简约优先

**可解决该问题的最简代码，不含任何推测性内容。**

- 不实现需求以外的额外功能。
- 一次性使用的代码不做抽象封装。
- 不添加需求未要求的灵活性与可配置设计。
- 无需针对不可能出现的场景编写异常处理逻辑。
- 若代码写了200行却可精简至50行，就重构优化。

扪心自问：“资深工程师会不会觉得这套方案过于繁琐？”如果答案是肯定的，就进行精简优化。

## 3. Surgical Changes

**非必要不要动任何东西，只清理自己遗留的手脚。**

修改已有代码时需遵循以下规则：
- 不得擅自“优化”周边代码、注释或代码格式。
- 未出现故障的代码，不做重构处理。
- 沿用项目现有编码规范，即便自身写法与之不同。
- 若发现无关的废弃代码，仅备注说明，切勿直接删除。

当你的改动产生孤立冗余代码时：
- 删除因你的修改而不再被使用的导入项、变量与函数。
- 未经许可，不得删除原本就存在的无效代码。

测试要求：每一处修改的代码行都需要能直接溯源至用户提出的需求。

## 4. 目标驱动执行

**确定验收标准，循环执行直至核验通过。**

将任务转化为可核验的目标：
- “新增校验功能”→“针对非法入参编写测试用例，再保证用例运行通过”
- “修复故障”→“编写可复现该故障的测试用例，再保证用例运行通过”
- “重构某模块”→“确保重构前后所有测试用例均可正常通过”

针对多步骤任务，请简述方案：
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

完善的成功标准能支撑你独立推进迭代；模糊的标准（如“能用就行”）则需要反复沟通确认。
